PHASE 3 — DETECTION ENGINE
==================================================
Tujuan: Rules + XGBoost berjalan, bisa generate alert dengan risk score.
File: backend/detection/rules.py, features.py, model.py

[x] 3.1  detection/rules.py → check_typology_rules(G, df)
         7 typologi dengan parameter set berbeda:
         - JUDOL: in_degree > 50 dalam 7 hari + amount < Rp500K + timing malam
         - SCAM: satu transfer besar masuk → langsung dipecah ke banyak rekening
         - DORMANT: tidak aktif > 90 hari → tiba-tiba aktif > 10 transaksi/3 hari
         - VENDOR_CANGKANG: dana dari rekening korporat → banyak rekening individu
         - BENDAHARA: pola mirip vendor cangkang tapi dari rekening pemerintah
         - QRIS_RING: banyak merchant_id berbeda → settlement ke satu rekening
         - PEP: transaksi ke/dari rekening terhubung PEP via perantara
         Return: list of {account_id, typology, rule_triggered, confidence}

[x] 3.2  detection/features.py → extract_features(G, df)
         Features per node:
         - in_degree, out_degree, degree_ratio
         - in_amount_sum, out_amount_sum, amount_ratio
         - unique_counterparts (in + out)
         - avg_tx_interval (detik antar transaksi)
         - shared_device_count (berapa akun pakai device sama)
         - burst_score (transaksi dalam 1 jam / total transaksi)
         - dormant_days (hari tidak aktif sebelum burst)
         - night_tx_ratio (% transaksi jam 00:00-04:00)
         - degree_centrality, pagerank (via Neo4j GDS query)
         Return: DataFrame satu baris per node

[x] 3.3  detection/model.py → train_xgboost(features_df, labels)
         - Input: features DataFrame + label is_laundering
         - Train/test split 80/20, stratified
         - Handle imbalance: scale_pos_weight
         - Simpan model ke models/xgboost_v1.pkl
         - Print: PR-AUC, F1, Precision, Recall @ threshold 0.7

[x] 3.4  detection/model.py → predict(features_df)
         - Load model dari pkl
         - Return: DataFrame {account_id, risk_score, risk_level}
         - risk_level: HIGH (>0.8), MEDIUM (0.5-0.8), LOW (<0.5)

[x] 3.5  detection/alerts.py → generate_alerts(predictions, rules_output)
         - Combine XGBoost score + rules output → final risk score
         - Threshold: risk_score > 0.7 → INSERT ke tabel alerts
         - Alert record: account_id, typology, risk_score, rule_triggered,
           cluster_id, status=NEW, created_at
         - Verifikasi: SELECT COUNT(*) FROM alerts;

PHASE 4 — TGN (TEMPORAL GRAPH NETWORK)
==================================================
Tujuan: Train TGN, jalankan ablation study, bandingkan semua model.
File: backend/ml/tgn_dataset.py, tgn_model.py, train_tgn.py

CATATAN: Training pakai HI-Large, demo pakai LI-Large (lihat E2 revisi notes)

[x] 4.1  ml/tgn_dataset.py → konversi ke PyG TemporalData
         - Node features: sama dengan XGBoost (extract_features output)
         - Edge index: from_account → to_account (integer encoded)
         - Edge attr: amount (normalized), channel (encoded), timestamp (unix)
         - Labels: is_laundering per node
         - Temporal split:
           train = 60% pertama (by timestamp)
           val = 20% berikutnya
           test = 20% terakhir (bukan random split!)

[x] 4.2  ml/tgn_model.py → definisi TGN
         - Pakai torch_geometric.nn.models.TGN
         - Memory module: track historical state per node
         - Message passing: update memory dari interaksi baru
         - Classifier head: linear layer → sigmoid → risk score
         - Loss: BCEWithLogitsLoss + class weight untuk imbalance

[x] 4.3  ml/train_tgn.py → training loop
         - Optimizer: Adam lr=0.0001
         - Epochs: 50, early stopping patience=10
         - Monitor: PR-AUC pada val set setiap epoch
         - Save checkpoint: models/tgn_v1.pt (best val PR-AUC)
         - Log training ke results/tgn_training_log.csv

[x] 4.4  ml/graphsage_baseline.py → GraphSAGE baseline
         - Pakai PyG GraphSAGE sebagai baseline comparison
         - Same features, same split sebagai TGN
         - Tujuan: ablation study bukan produksi

[x] 4.5  Ablation study → results/ablation_results.csv
         Bandingkan semua model pada test set:
         Rules only | XGBoost tabular | XGBoost+graph features |
         GraphSAGE | TGN | Ensemble (XGBoost+TGN)
         Metrik: PR-AUC, F1@0.7, Precision@100, Recall@0.7
         Update angka di proposal setelah benchmark selesai

[x] 4.6  ml/ensemble.py → Ensemble XGBoost + TGN
         - Weighted average: XGBoost 0.4 + TGN 0.6 (atau tune di val)
         - Ini adalah production model yang dipakai di demo
         - Save: models/ensemble_v1.pkl

[x] 4.7  Demo data preparation
         - Load LI-Large (bukan HI-Large) ke PostgreSQL sebagai demo_transactions
         - Generate alerts dari LI-Large dengan ensemble model
         - Seed 5 kasus pre-loaded: judol ring, scam, dormant, PEP, vendor cangkang

PHASE 5 — API LAYER (gRPC INGESTION + FASTAPI REST)
==================================================
Tujuan: Dua protokol, dua peran:
  - gRPC: high-throughput ingestion transaksi dari sistem bank
  - FastAPI REST: operasi dashboard, query graph, case, alert
File: backend/grpc/ + backend/api/routes/ + sub-routers

CATATAN ARSITEKTUR PROTOKOL:
- gRPC dipilih untuk INGESTION karena binary protobuf, throughput
  tinggi, bidirectional streaming — cocok terima jutaan transaksi
  dari core banking. Flow: bank → gRPC stream → Kafka → consumer.
- FastAPI REST untuk FRONTEND karena browser-friendly (browser
  tidak native gRPC tanpa proxy grpc-web).
- MVP/demo: ingestion via Kafka producer (replay/simulate) sudah
  jalan; gRPC server adalah gateway produksi saat konek bank nyata.

[ ] 5.0  grpc/ — Ingestion Gateway (gRPC)
         - proto/transaction.proto: definisi message TransactionEvent
           + service IngestionService (rpc StreamTransactions stream)
         - grpc/server.py: terima gRPC stream → validasi → publish
           ke Kafka topic transactions.raw (jembatan ke pipeline existing)
         - grpc/client_example.py: contoh client bank push transaksi
         - Unary + server-streaming + client-streaming untuk batch
         - MVP: jalan paralel dengan Kafka producer; produksi:
           gantikan producer sebagai pintu masuk utama dari bank

[ ] 5.1  api/routes/graph.py
         GET  /graph/overview → stats (nodes, edges, clusters, density)
         GET  /graph/cluster/{cluster_id} → nodes + edges dalam cluster
         GET  /graph/node/{account_id}/neighbors → tetangga 2 hop
         GET  /graph/node/{account_id}/ppr → PPR score dari node ini
         GET  /graph/node/{account_id}/flags → transaction flags
         POST /graph/sweep/{account_id} → trigger retrospective sweep
         GET  /graph/institutions → daftar institusi + stats per institusi

[ ] 5.2  api/routes/alerts.py
         GET   /alerts → list alert (filter: status, typology, risk_score,
               institution_id, date_from, date_to)
         GET   /alerts/{alert_id} → detail satu alert
         PATCH /alerts/{alert_id}/status → NEW|IN_REVIEW|CONFIRM|FP
         POST  /alerts/{alert_id}/assign → assign ke analyst

[ ] 5.3  api/routes/cases.py
         POST  /cases → buat case dari alert_id
         GET   /cases/{case_id} → detail case + audit trail
         PATCH /cases/{case_id} → update notes, status, assigned_to

[ ] 5.4  api/routes/reports.py
